# PRD 需求文档一键生成



> 一句话描述需求 → 7模块完整PRD → 可直接交付开发



## 核心能力



- 从模糊需求自动结构化为完整PRD

- 用户故事格式（As a / I want / So that）

- BDD 验收标准（Given-When-Then）

- 自动推导数据模型和接口设计

- 风险识别和灰度策略



---



## 边界条件与异常处理



| 场景 | 识别条件 | 处理策略 |

|------|----------|----------|

| 模糊输入 | 用户仅提供1句话，无上下文 | 启动结构化澄清：提出3个关键问题（目标用户、核心操作、成功指标），不直接生成 |

| 需求冲突 | 检测到互斥要求（如"简单易用"+"覆盖所有场景"） | 明确列出冲突点，请用户排定优先级后再继续 |

| 范围过大 | 单个PRD超过15个用户故事或跨3个以上子系统 | 建议拆分为多期/多PRD，先确认分期策略再生成当期内容 |

| 输出不满意 | 用户反馈PRD不符合预期 | 不重新全量生成，而是询问"哪个模块需要调整？方向是什么？"，定向修订 |

| 缺少领域知识 | 涉及专业领域（医疗/金融/法规等）且无足够上下文 | 明确告知知识盲区，要求用户补充业务规则或参考文档 |

## 执行流程



### Step 1: 需求澄清



收到需求后，先快速澄清（不超过 3 个问题）：

- 这个功能解决什么业务问题？目标用户是谁？

- 核心操作路径是什么（用户怎么用）？

- 有没有参照产品或已有设计稿？



> 如果用户提供的信息足够（含截图、原型链接、详细描述），跳过澄清直接生成。



#### 🔲 检查点 1：需求理解确认



**在生成PRD之前，必须执行此检查点：**



将理解到的需求复述给用户，格式如下：



```

📋 需求理解确认：

- 产品/功能：[名称]

- 目标用户：[角色]

- 核心场景：[1-2句描述]

- 成功指标：[预期效果]

- 约束条件：[技术/时间/资源限制]



以上理解是否正确？确认后开始生成PRD。

```



**等待用户确认后再进入 Step 2。**



### Step 2: 生成 PRD（7模块标准结构）



#### 🔲 检查点 2：PRD 大纲确认



完成7模块初稿后，先输出 PRD 概要摘要供用户审核：



```

📄 PRD 大纲预览：

- 用户角色：X 个（列出）

- 功能点：X 个用户故事（列出 US 编号和标题）

- 数据实体：X 个（列出核心实体名）

- 上线策略：X 阶段灰度



方向是否正确？如需调整范围/侧重点请告知，确认后输出完整内容。

```



**等待用户确认方向正确后，再输出完整7模块详细内容。**

## PRD 标准结构



### 模块一：背景与目标



| 字段 | 内容 |

|------|------|

| 项目名称 | [自动提取] |

| 需求来源 | [用户输入 / 业务方反馈 / 竞品分析 / 数据驱动] |

| 业务目标 | 1-3 个可量化目标 |

| 核心指标 | 北极星指标 + 2-3 个过程指标 |

| 优先级 | P0(必做) / P1(应做) / P2(可做) |



### 模块二：用户角色与场景



为每个角色创建画像：



```

角色：[角色名]

背景：[职责、技术水平、使用频率]

核心诉求：[要解决什么问题]

典型场景：[在什么情况下使用、做什么操作、期望什么结果]

```



### 模块三：功能需求



每个功能用用户故事格式：



```

## US-001: [功能名称]

优先级: P0



As a [角色],

I want [具体功能],

So that [业务价值].



### 验收标准



AC-1: 

  Given [前置条件]

  When [用户操作]

  Then [预期结果]



AC-2:

  Given [另一个前置条件]  

  When [用户操作]

  Then [预期结果]



### 业务规则

- BR-1: [具体规则]

- BR-2: [具体规则]



### 异常流程

- EX-1: [异常情况] → [处理方式]

```



### 模块四：非功能需求



| 维度 | 要求 | 验收标准 |

|------|------|----------|

| 性能 | 页面加载 / 接口响应 | P95 < Xms |

| 安全 | 认证、授权、数据加密 | [具体要求] |

| 兼容性 | 浏览器、设备、分辨率 | [覆盖范围] |

| 可用性 | SLA 目标 | 99.X% |

| 可扩展性 | 预估用户量 / 数据量增长 | [容量规划] |



### 模块五：信息架构与交互流程



- 页面层级结构（树状图）

- 核心操作流程（步骤图）

- 页面跳转关系（导航图）

- 关键交互说明（状态流转）



### 模块六：数据需求



**数据模型：**



```

实体: [实体名]

├── 字段1: 类型 [必填/选填] — 说明

├── 字段2: 类型 [必填/选填] — 说明

└── 关联: → [关联实体] (1:N / N:N)

```



**接口设计（核心接口）：**



| 接口 | 方法 | 路径 | 入参 | 出参 | 说明 |

|------|------|------|------|------|------|



### 模块七：上线计划与灰度策略



| 阶段 | 范围 | 时间 | 回滚条件 |

|------|------|------|----------|

| 灰度1 | 内部团队 | X天 | [指标阈值] |

| 灰度2 | X%用户 | X天 | [指标阈值] |

| 全量 | 100% | - | - |



**风险识别：**



| 风险 | 概率 | 影响 | 缓解措施 |

|------|------|------|----------|

## 质量检查清单



- [ ] 每个功能都有用户故事格式（As a/I want/So that）

- [ ] 每个用户故事至少 2 条验收标准（Given-When-Then）

- [ ] 验收标准可测试、无歧义

- [ ] 非功能需求有量化指标

- [ ] 数据模型覆盖核心实体

- [ ] 核心接口已定义

- [ ] 灰度策略含回滚条件

## 参考资源



- PRD 结构参考：可结合团队已有 PRD 模板（如飞书模板库、Confluence 模板）进行风格统一

- 验收标准写法：参考 BDD (Behavior-Driven Development) 的 Gherkin 语法规范

- 用户故事拆分：参考 INVEST 原则（Independent, Negotiable, Valuable, Estimable, Small, Testable）

- 非功能需求量化：参考 ISO 25010 软件质量模型（功能适合性、性能效率、兼容性、可用性、可靠性、安全性、可维护性、可移植性）

**不适合：** Bug 修复工单、纯技术方案文档
