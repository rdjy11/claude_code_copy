# Darwin 8 维度 Rubric 详细评分标准



## 评分体系总览



满分 100 分 = 结构维度 60 分 + 效果维度 40 分



| 维度 | 代号 | 满分 | 类别 | 核心问题 |

|------|------|------|------|---------|

| Frontmatter 质量 | D1 | 8 | 结构 | Agent 能否自动发现和识别这个 Skill？ |

| 工作流清晰度 | D2 | 15 | 结构 | Agent 读完后是否知道第一步做什么、第二步做什么？ |

| 边界条件覆盖 | D3 | 10 | 结构 | 异常情况发生时 Agent 是否知道怎么处理？ |

| 检查点设计 | D4 | 7 | 结构 | 关键决策点是否强制暂停获取用户确认？ |

| 指令具体性 | D5 | 15 | 结构 | 指令是否具体到 Agent 无需猜测即可执行？ |

| 资源整合度 | D6 | 5 | 结构 | 是否合理使用 references/scripts/ 分离关注点？ |

| 整体架构 | D7 | 15 | 效果 | 技能的整体设计是否优雅、模块化、可维护？ |

| 实测表现 | D8 | 25 | 效果 | Agent 实际执行时是否能产出高质量结果？ |



---



## D1: Frontmatter 质量（满分 8 分）



评估 YAML frontmatter 是否完整、准确、有利于自动发现。



| 分数 | 标准 |

|------|------|

| 8 | name + description（含关键词+触发词）+ version + metadata（author/created） |

| 7 | name + description（含触发词）+ 部分 metadata |

| 6 | name + description（通用描述，无明确触发词） |

| 5 | name + description（过于简短或模糊） |

| 4 | 仅有 name 和极简 description |

| ≤3 | 缺失 frontmatter 或格式不合规 |



**关键检查项**：

- name 是否符合命名规范（小写+连字符）

- description 是否包含领域关键词（用于 agent 自动匹配）

- 是否有触发词/激活条件说明

- 是否有 version 字段



---



## D2: 工作流清晰度（满分 15 分）



评估 Agent 能否从头到尾明确知道每一步该做什么。



| 分数 | 标准 |

|------|------|

| 15 | 完整编号步骤/checklist + 流程图(dot/mermaid) + 状态转换明确 + 终止条件清晰 |

| 13-14 | 有清晰步骤但缺少可视化流程图，或步骤间转换条件不够明确 |

| 11-12 | 有步骤描述但顺序或依赖关系不清晰，需要 Agent 自行推断 |

| 9-10 | 有方法论框架但未转化为可执行步骤 |

| ≤8 | 描述性文字为主，Agent 需要自行设计执行路径 |



**关键检查项**：

- 是否有编号步骤或 checklist

- 步骤之间是否有明确的前后依赖

- 是否有条件分支（if/then）

- 终止条件是否明确（何时算"完成"）

- 是否有流程可视化（dot/mermaid/ASCII）



---



## D3: 边界条件覆盖（满分 10 分）



评估异常场景的处理是否充分。



| 分数 | 标准 |

|------|------|

| 9-10 | 覆盖 5+ 种异常（输入不足/工具失败/输出不符/用户拒绝/超时等）+ 每种有具体处理方案 |

| 7-8 | 覆盖 3-4 种异常 + 有处理方案 |

| 5-6 | 有"何时不使用"的说明 + 1-2 种异常处理 |

| 3-4 | 仅有简单的"适用/不适用"声明 |

| ≤2 | 无边界条件说明 |



**关键检查项**：

- 输入不足或无效时如何处理

- 依赖工具/API 失败时的降级方案

- 输出质量不达标时的修复循环

- 用户中途拒绝或改变方向时的处理

- "何时不使用此技能"的明确声明



---



## D4: 检查点设计（满分 7 分）



评估关键决策点是否强制暂停获取用户确认。



| 分数 | 标准 |

|------|------|

| 7 | 3+ 个显式检查点，使用"必须暂停"/"HARD-GATE"等强制语言 + 每个检查点有具体确认清单 |

| 5-6 | 2 个显式检查点 + 有确认内容说明 |

| 3-4 | 1 个检查点或有"建议确认"但非强制 |

| 1-2 | 仅有隐含的确认时机，无强制暂停 |

| 0 | 无任何检查点设计 |



**关键检查项**：

- 是否有"必须暂停"/"MUST stop"/"HARD-GATE" 等强制性语言

- 检查点是否明确说明需要确认什么

- 是否覆盖"启动前确认"和"输出后确认"两个关键时机

- 检查点是否放在不可逆操作之前



---



## D5: 指令具体性（满分 15 分）



评估指令是否具体到 Agent 无需猜测即可执行。



| 分数 | 标准 |

|------|------|

| 15 | 精确到文件路径/代码模板/数值标准/格式规范，Agent 零歧义执行 |

| 13-14 | 大部分指令具体，少量需 Agent 自行判断 |

| 11-12 | 有具体示例但非所有场景都覆盖 |

| 9-10 | 指令明确方向但缺乏执行细节 |

| ≤8 | 抽象原则为主，Agent 需自行转化为操作 |



**关键检查项**：

- 是否有代码/模板示例

- 是否有精确的格式要求（如"卡片圆角12px"）

- 是否有 Good/Bad 对比示例

- 输出文件路径是否明确

- 是否用了"适当的"/"合理的" 等模糊词（扣分项）



---



## D6: 资源整合度（满分 5 分）



评估是否合理使用外部文件分离关注点。



| 分数 | 标准 |

|------|------|

| 5 | 有 references/ + scripts/ + 模板文件，SKILL.md 控制在 300 行以内，关注点清晰分离 |

| 4 | 有 references/ 或 scripts/ 之一，主文件适度精简 |

| 3 | 无外部文件但主文件行数合理（<300行）且功能单一 |

| 2 | 无外部文件，主文件 300-500 行，部分内容应外置 |

| 1 | 无外部文件，主文件 >500 行，严重堆积 |



**关键检查项**：

- SKILL.md 行数是否合理

- 方法论/知识库内容是否外置到 references/

- 可执行脚本是否放在 scripts/

- 模板文件是否独立存放

- 是否有重复内容可以提取



---



## D7: 整体架构（满分 15 分）



评估技能的整体设计质量。



| 分数 | 标准 |

|------|------|

| 15 | 模块化清晰 + 可扩展 + 与其他技能协作关系明确 + 命名规范 + 单一职责 |

| 13-14 | 架构良好但有小瑕疵（如协作关系未说明） |

| 11-12 | 结构合理但模块划分不够清晰 |

| 9-10 | 功能堆叠，缺乏层次感 |

| ≤8 | 结构混乱或职责不单一 |



**关键检查项**：

- 是否遵循单一职责原则

- 是否有清晰的"何时使用/何时不使用"边界

- 是否说明与其他 skill 的配合关系

- 文件组织是否规范

- 是否可独立使用（不强制依赖其他 skill）



---



## D8: 实测表现（满分 25 分）



评估 Agent 实际执行时的预期效果（dry_run 模式下基于推演）。



| 分数 | 标准 |

|------|------|

| 23-25 | Agent 可完全自主执行，输出质量专业级，有验证机制确保质量 |

| 20-22 | Agent 大部分可自主执行，少量环节需人工介入 |

| 17-19 | Agent 能执行但可能偏离预期，需要较多人工校正 |

| 14-16 | Agent 执行困难，更像是参考手册而非可执行指令 |

| ≤13 | Agent 读完后仍不知道如何行动 |



**关键评估维度**：

- **自主性**：Agent 能否在最少人工介入下完成任务

- **输出质量**：预期产出是否达到专业水准

- **可验证性**：是否有内建的质量检查机制

- **适用范围**：是否适用于目标用户群体（语言、平台、领域）

- **社区验证**：Star数/下载量/实际使用反馈（如有）

## 评分原则



1. **证据驱动**：每个分数必须有文本证据支撑，不可凭感觉

2. **相对一致性**：同一评审批次内，相同表现应得相同分数

3. **功能优先**：实际效果(D7+D8=40分)权重高于形式完美

4. **不惩罚简洁**：短小精悍但完整 > 冗长全面但松散

5. **评框架不评语言**：英文/中文不影响结构分，仅在 D8 适用性中体现
