# Grill Me — 需求追问引擎



逐分支深入追问用户的计划或设计，直到所有决策路径都被解析清楚。



## 执行规则



1. **逐分支深入**：沿决策树每条路径追问，一次只解决一个分支

2. **每问附推荐答案**：用户可直接说"yes"/"对"加速通过

3. **能查代码就查代码**：涉及"已有什么"时，自动读代码回答，不问用户

4. **一次问 1-2 个问题**：不要一次性抛出 5 个问题轰炸用户



## 追问维度覆盖



按以下维度逐步展开（不要求全部覆盖，按相关性选择）：



| 维度 | 典型问题 |

|------|---------|

| Happy Path | 正常流程的完整步骤？ |

| 状态流转 | 提交后状态如何变化？能否撤回？ |

| 空状态 | 无数据时显示什么？首次使用引导？ |

| 边界情况 | 名称超长？数量极端？必填为空？ |

| 跨模块联动 | 这里的操作影响其他模块吗？ |

| 权限差异 | 不同角色看到的内容/操作有别吗？ |

| 错误处理 | 失败时如何提示？ |



## CQ 驱动验证（系统设计类任务必做）



当追问对象是**系统/产品设计**时，在摘要前额外产出 CQ 列表：



1. 从已确认的需求中提取 8-15 个**胜任力问题**（系统必须能回答的核心问题）

2. 每个 CQ 标注：涉及实体、推理类型、优先级（P0/P1/P2）

3. 用 CQ 反向验证数据模型：每个 P0 CQ 是否有对应的实体+关系支撑？

4. 输出缺口表：CQ 需要但模型中不存在的实体/关系



**CQ 三要求**：可回答（系统能用数据给出答案）、有业务价值（对应用户决策）、可验证（答案正确性可测）



**输出格式**（嵌入摘要的"关键决策"之后）：



```

### CQ 列表（系统能力边界）



| CQ# | 问题 | 涉及实体 | 推理类型 | 优先级 |

|-----|------|---------|---------|--------|

| CQ1 | ... | ... | 分类/检索/规则/关联 | P0 |



### 数据模型验证



| CQ# | 能否支撑 | 缺口 |

|-----|---------|------|

```



跳过条件：纯 UI 改动、bug 修复、非系统设计类需求。



---



## 终止条件



当以下**任一**条件满足时，停止追问并进入摘要输出：



1. 所有已识别的决策分支都有明确结论

2. 用户说"够了"/"可以开始了"/"不用再问了"

3. 连续 3 个问题用户回答"你决定"/"随便"/"默认就行"（说明细节不重要）

4. 追问轮次超过 8 轮且核心路径已覆盖



## 摘要输出模板



追问结束后，输出结构化摘要：



```

## 需求共识摘要



### 目标

{一句话描述要达成什么}



### 范围

- {涉及的页面/模块/组件}



### 关键决策

| # | 决策点 | 结论 | 备注 |

|---|--------|------|------|

| 1 | ... | ... | ... |



### 约束

- {必须遵守的规则}



### 不做

- {明确排除的部分}



### 验收标准

- [ ] {可操作的检查步骤 1}

- [ ] {可操作的检查步骤 2}

```



用户确认摘要后，方可进入实现阶段。
